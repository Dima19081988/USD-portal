# Требования безопасности

## Общий принцип

Безопасность системы является обязательной частью архитектуры проекта с первого
этапа. Все ключевые решения по аутентификации, авторизации, сессиям, логированию
и хранению данных должны приниматься с учетом secure-by-default подхода.

## Основные требования

- у каждого пользователя должен быть уникальный аккаунт;
- общие учетные записи запрещены;
- доступ должен строиться по принципу минимально необходимых прав;
- все чувствительные действия должны логироваться;
- проверка прав должна происходить на backend, а не только на frontend.

## Аутентификация

- хранение паролей только в виде хеша;
- использование Argon2 или bcrypt;
- защита от brute force;
- ограничение количества попыток входа;
- возможность добавить MFA в будущем.

## Сессии и токены

- безопасное завершение сессий;
- access token и refresh token при JWT-подходе;
- refresh token rotation;
- HttpOnly cookies;
- Secure cookies;
- SameSite cookies;
- ограничение по времени жизни сессии.

## Защита данных

- использование HTTPS/TLS;
- безопасное хранение секретов вне репозитория;
- исключение `.env` из git;
- защита резервных копий;
- ограничение доступа к продакшн-конфигурации.

## Аудит

Должны логироваться:
- успешные входы;
- неуспешные входы;
- выходы из системы;
- изменение ролей и прав;
- изменение шаблонов;
- изменение материалов и контента;
- изменение критичных настроек системы.

## Безопасность backend

- валидация входных данных;
- централизованный error handling;
- rate limiting;
- защита чувствительных endpoint;
- контроль доступа через permissions;
- запрет доверять только данным с frontend.

## Безопасность разработки

- `.env` и секреты не должны попадать в git;
- production и development конфигурации должны быть разделены;
- зависимости должны обновляться регулярно;
- security review должен проводиться хотя бы на базовом уровне до деплоя.
